배운점#

반복적인 계산이 필요한 데이터는 따로 테이블을 만들어 주기적으로 업데이트 하자.#

프로젝트를 진행할때 모델링에 시간을 가장 많이 투자했다. 금융투자관련 웹사이트다 보니, 사용자가 투자한 금액을 한눈에 보여주는 투자요약페이지가 있었다.

사용자가 투자한 각 상품별로 매달 계획된 이자,세금,수수료가 기록된 user_paybacks 라는 테이블이 있어서, 투자요약 data를 요청할 때마다 지급이 완료된 회차와 미지급된 회차를 따로 분류하여 계산하였다.

하지만 이자가 지급되는 것이 실시간으로 바뀌는 것이 아니기 때문에, 각 상품별로 이자가 납부될때 마다 업데이트를 해주는 테이블을 만들었다면 사용자가 매번 투자요약을 요청할때 복잡한 계산을 반복하지 않고 미리 계산된 테이블의 data를 가져와 컴퓨팅 리소스를 줄일 수 있다는 것을 깨달았다.

 

상태값과 같이 변경이 없는 테이블은 Enum을 활용하되, Database만으로도 모델링 구조를 알 수 있도록 만들자.#

상품의 상환상태(정상, 연체 등)나 신용등급(A+, C- 등)과 같은 테이블이 너무 많아서 ERD의 테이블 수를 줄이고자, 상품 model 안에 기존 Foreign Key들을 아래와 같이 Enum으로 관리했다.

class Deal(TimeStampModel):
    class Category(models.IntegerChoices):
        MORTGAGE = 1, '부동산 담보대출' 
        CREDIT   = 2, '개인신용'
        SPECIAL  = 3, '스페셜딜'
        COMPANY  = 4, '기업'
        ETC      = 5, '기타'

    class Grade(models.IntegerChoices):
        A_PLUS  = 1, 'A+'
        A       = 2, 'A'
        A_MINUS = 3, 'A-'
        B_PLUS  = 4, 'B+'
        B       = 5, 'B'
        B_MINUS = 6, 'B-'
        C_PLUS  = 7, 'C+'
        C       = 8, 'C'
        C_MINUS = 9, 'C-'
        D_PLUS  = 10, 'D+'
        D       = 11, 'D'
        D_MINUS = 12, 'D-'

    class Status(models.IntegerChoices):
        APPLYING              = 1, '신청중' 
        NORMAL                = 2, '정상'
        DELAY                 = 3, '상환지연'
        OVERDUE               = 4, '연체'
        NONPERFORM            = 5, '부실'
        NORMAL_COMPLETION     = 6, '정상상환완료'
        NONPERFORM_COMPLETION = 7, '부실상환완료'
        SCHEDULED             = 8, '모집예정'

    class RepaymentMethod(models.IntegerChoices):
        MIX             = 1, '혼합'
        EQUAL_SUM       = 2, '원리금균등'
        MATURE          = 3, '만기상환'
        EQUAL_PRINCIPAL = 4, '원금균등'

    name             = models.CharField(max_length=100)
    category         = models.IntegerField(choices=Category.choices)
    grade            = models.IntegerField(choices=Grade.choices)
    earning_rate     = models.DecimalField(max_digits=4, decimal_places=2)
    interest_rate    = models.DecimalField(max_digits=4, decimal_places=2)
    repayment_period = models.IntegerField()
    repayment_method = models.IntegerField(choices=RepaymentMethod.choices)
    net_amount       = models.IntegerField()
    repayment_day    = models.IntegerField()
    start_date       = models.DateField()
    end_date         = models.DateField(null=True)
    reason           = models.CharField(max_length=100)
    debtor           = models.ForeignKey('Debtor', on_delete=models.PROTECT)
    status           = models.IntegerField(choices=Status.choices)

    class Meta:
        db_table = 'deals'

예를 들어, Category=1 인 상품은 테이블을 참조하지 않더라도 바로 부동산 상품인 것이다.

이러한 방법은 ERD내의 테이블 수를 15개에서 10개로 줄일 수 있었고, View로직에서도 정참조하는 횟수도 줄일 수 있었다.

하지만, 이러한 방법은 Django코드 내부를 들여다 봐야지만 Category=1이 어떤 의미인지 알 수 있다. 백엔드 개발시에, Database는 공동으로 사용하고 백엔드 개발은 Django, Node.js 등 다양하게 개발하는 경우도 있기 때문에 Database만 보고도 모델링 구조를 알 수 있어야 한다.

따라서, Enum을 사용하되 모델 내부에서 사용한다면 Category=1이 아닌 Category=mortgage가 될 수 있도록 TextChoice를 사용하거나, ForeignKey를 활용해야 한다.

 

팀원의 진행상황을 정확히 파악하자#

1차 프로젝트를 한번 진행하고 나서, 프로젝트의 로드맵을 칸반보드로 관리하는데에는 좀 더 익숙해진 상태였다.

하지만, 프로젝트 To-Do를 나눌 때 개인이 각자 해야할 일을 작성하다 보니, 매일 미팅에서 진행상황을 공유할 때 어제 한일이 프로젝트의 어느정도 규모의 일이고 더 중요한 일이 밀려있는지를 파악하지 못했다.

내 업무도 처리하기 바쁘다 보니, 다른 사람들의 진행상황을 파악하지 못한 것이다. 내 업무가 바쁘더라도, 매일 미팅 전후로 다른 사람들이 어떠한 일을 하고 있고 어디서 막히는지 좀더 세밀하게 관찰할 필요가 있다고 느꼈다.
